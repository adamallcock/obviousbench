# Opus 4.8 Adaptive-Thinking Anomaly — Root-Cause Diagnosis

## TL;DR

The Opus 4.8 "low thinking / low cost" pattern is **mostly expected provider
behavior, exaggerated by two local measurement bugs.** There is no broken request
shape. Specifically:

1. **Provider behavior (expected).** Adaptive thinking treats `effort` as *soft
   guidance, not a floor*. Opus 4.8 is well-calibrated and declines to think on
   obvious prompts even at `effort=max`. ObviousBench is deliberately full of
   obvious prompts, so Opus skips thinking on almost all of them. This is a
   feature (low overthinking), not a defect.
2. **Telemetry bug (local).** Inspect's `ModelUsage.reasoning_tokens` for Claude
   4.x is the **token count of the returned *summary* text**, not billed thinking.
   The canonical billed field is `usage.output_tokens_details.thinking_tokens`,
   which Inspect's Anthropic provider does not read. The summary count is so
   decoupled from reality that it can *exceed* total `output_tokens`.
3. **Cost bug (local).** The cost pricer carves `output_text = output_tokens −
   reasoning_tokens` and prices the carved-out reasoning separately. The runcost
   `claude-opus-4-8:models-dev` price card has **no reasoning rate**, so runcost
   silently drops that component → Opus output is **undercharged ~36%** at `max`.
   Sonnet uses a litellm card that *does* price the reasoning bucket, so its cost
   is ~correct; this is why only Opus looks "too cheap."

Net: the Opus-vs-Sonnet thinking/cost gap is real in direction (Opus thinks less)
but **overstated in magnitude** by (2) and (3). Do not publish Opus thinking
curves or cost-per-correct from this run without re-pricing and re-instrumenting.

## 1. What we verified against live Anthropic docs (primary sources)

Fetched 2026-06-03 from `platform.claude.com`:

- `/docs/en/build-with-claude/adaptive-thinking`
- `/docs/en/build-with-claude/effort`
- `/docs/en/build-with-claude/extended-thinking`

Key verbatim findings:

- **Adaptive = optional thinking, model decides.** "In adaptive mode, thinking is
  optional for the model. Claude evaluates the complexity of each request and
  determines whether and how much to use extended thinking. At the default effort
  level (`high`), Claude almost always thinks. At lower effort levels, Claude may
  skip thinking for simpler problems."
- **Effort is a soft signal, not a budget/floor.** "Effort is a behavioral
  signal, not a strict token budget. At lower effort levels, Claude will still
  think on sufficiently difficult problems, but it will think less than it would
  at higher effort levels for the same problem."
- **Adaptive can skip thinking entirely.** "When using `thinking.type:
  "adaptive"` and the model skips thinking for a simple request, no thinking block
  is produced regardless of `display`."
- **Opus 4.8 defaults `display` to `omitted`.** "On Claude Opus 4.8, Claude Opus
  4.7, and Claude Mythos Preview, `display` defaults to `"omitted"`… you must set
  `display: "summarized"` explicitly to receive summarized thinking." (Inspect
  *does* set `summarized`, so this is not the cause — see §3.)
- **Canonical billed-thinking field.** "To see how many billed output tokens were
  spent on internal reasoning, read `usage.output_tokens_details.thinking_tokens`
  in the response. This value reflects the raw reasoning the model generated (not
  the summarized text returned in the body)… `output_tokens` remains the
  inclusive, authoritative total used for billing."
  ```json
  { "usage": { "input_tokens": 25, "output_tokens": 348,
               "output_tokens_details": { "thinking_tokens": 312 } } }
  ```
- **Summaries are produced by a different model and are not the billed thinking.**
  "You're charged for the full thinking tokens generated by the original request,
  not the summary tokens. The billed output token count will not match the count
  of tokens you see in the response… On Claude 4 models, the first few lines of
  thinking output are more verbose." (This is why an Opus *summary* can be longer
  than its tiny *billed* thinking — see §2 smoke.)
- **No beta header is required** for adaptive thinking or for `effort`.
  `output-128k-2025-02-19` is only about raising the output-token cap above the
  model default (Inspect adds it whenever `max_tokens > 8192`); it is unrelated to
  enabling thinking.

## 2. Direct live smoke (below Inspect, below ObviousBench)

`scripts/smoke_anthropic_thinking.py`, key read from Keychain
(`security find-generic-password -s ANTHROPIC_API_KEY -w`), never printed. Request
shape held constant:
`thinking={type:adaptive, display:summarized}`, `output_config={effort:max}`,
`max_tokens=1500`.

| model | prompt | content blocks | output_tokens | `output_tokens_details.thinking_tokens` (billed) | count_tokens(summary) (what Inspect records) |
|---|---|---|---:|---:|---:|
| claude-opus-4-8 | trivial (`9+14-1`) | text | 3 | **0** | 0 |
| claude-opus-4-8 | harder (3-switch puzzle) | thinking,text | 225 | **23** | **63** |
| claude-sonnet-4-6 | trivial | thinking,text | 32 | 26 | 21 |
| claude-sonnet-4-6 | harder | thinking,text | 403 | 232 | 222 |

Reads:

- Opus 4.8 at `effort=max` produces **no thinking block and 0 billed thinking** on
  a trivial prompt (expected; adaptive skipped it).
- On the harder prompt Opus billed only **23** thinking tokens but its
  **summary re-tokenizes to 63** — the metric Inspect records is ~3× the truth and
  in the wrong direction. `output_tokens` (225) is the authoritative total and
  includes the 23 billed thinking tokens.
- Sonnet's summary ≈ its billed thinking (21 vs 26; 222 vs 232), which is why the
  Sonnet numbers *looked* sane and the Opus numbers did not.

## 3. How the benchmark stack actually behaves (code trace)

Request build — `inspect_ai/model/_providers/anthropic.py`:

- Panel `effort` → `reasoning_effort` (`model_panel_runner.py:_translate_control_style_generation_settings`).
- For frontier Claude (4.6+), `effort_from_reasoning_effort` (line ~1559) maps it
  and the provider sends `thinking={type:"adaptive", display:"summarized"}` +
  `output_config={effort:…}` (lines 844–849). **This request shape is correct and
  matches the docs.** `max` is preserved for Opus (`is_claude_frontier()` true);
  `xhigh` is preserved for 4.7/4.8. `output-128k-2025-02-19` is added only when
  `max_tokens>8192`. The benchmark path does **not** send `effort-2025-11-24`
  (that beta is only added when `config.effort` is set directly, which the panel
  does not do) — and per docs none is required.

Telemetry — same file, `model_output_from_message` (lines 2337–2344, 2433):

```python
reasoning_tokens = 0
for content_block in message.content:
    if isinstance(content_block, ThinkingBlock):
        reasoning_tokens += await count_tokens(client, model, content_block.thinking)
...
reasoning_tokens=reasoning_tokens if reasoning_tokens > 0 else None,
```

The comment at line 2699 confirms the source: "anthropic reasoning is now always a
summary." So `ModelUsage.reasoning_tokens = count_tokens(summary_text)`. It never
reads `usage.output_tokens_details.thinking_tokens`. (By contrast the OpenAI
Responses provider *does* read `output_tokens_details.reasoning_tokens` —
`openai_responses.py:221` — which is why OpenAI reasoning telemetry is trustworthy
and Anthropic's is not.)

`obviousbench/analysis/logs.py:77` ingests this value verbatim
(`reasoning_tokens=_usage_int(usage, "reasoning_tokens")`). Inspect's `ModelUsage`
carries no `output_tokens_details`, so **the true billed thinking is absent from
the `.eval` logs** — existing runs can be re-priced but cannot be re-instrumented
for a thinking axis without re-running.

Cost — `scripts/price_usage_with_runcost.mjs:65,80–81`:

```js
const outputTextTokens = Math.max(outputTokens - reasoningTokens, 0);
...
component("output_text_tokens", outputTextTokens),
component("output_reasoning_tokens", reasoningTokens),
```

`output_tokens` already includes billed thinking, so this split is only harmless
when both buckets price at the same rate. Evidence from this run's ledgers:

| | price card | output_text rate | output_reasoning component |
|---|---|---|---|
| Opus 4.8 | `anthropic:claude-opus-4-8:models-dev` | $25/M | **dropped (no reasoning rate on card)** |
| Sonnet 4.6 | `anthropic:claude-sonnet-4-6:litellm` | $15/M | priced at $15/M (nets out) |

Sample `obviousbench.char_count.en.v0.public.000008` ("How many r's in raspberry?",
Opus max): `output_tokens=63`, `reasoning_tokens=37` → priced as `output_text=26`
only; the 37 reasoning tokens were charged **$0**. Run totals: Opus max billed
`output=2427` but only `output_text=1177` was priced (1542 reasoning tokens at $0).

- Reported Opus max cost: **$0.054625**
- Correct (`input×$5 + output_tokens×$25`): **≈$0.0859**
- → **~36% undercharge**, entirely on the Opus rows.

Note the per-sample clamp `max(output−reasoning,0)` also masks the impossible cases
where the *summary count exceeds total billed output* (e.g. "Convert 3 miles":
`output=24`, `reasoning=37`) — proof the value is not a token subset of output.

Practical implications for us:
- **Opus 4.8 needs no different headers, betas, or request shape than Sonnet 4.6.**
  The only model-specific divergences are: `display` defaults to `omitted` (Inspect
  overrides to `summarized`), `xhigh` is Opus-4.7/4.8-only, and manual
  `budget_tokens` is rejected on Opus 4.7/4.8 (Inspect already avoids it for
  frontier models).
- `max_tokens` for `xhigh`/`max` should be generous (docs suggest ≥64k for
  agentic Opus work). Inspect's `max_tokens_for_config` already floors `xhigh`/`max`
  at 64k and caps Opus 4.6+ at 128k — fine for our short-answer task.

## 5. Recommendation for the benchmark / report

1. **Stop reporting Anthropic `reasoning_tokens` as thinking effort.** For Claude
   4.x it is summary length, not billed thinking, and is not comparable to OpenAI
   (which reports true reasoning tokens) or to itself across models. Either drop the
   column for Anthropic rows or relabel it "summary tokens (not billed thinking)."
2. **Re-price all Anthropic rows from `output_tokens`** (the authoritative billed
   total) before any cost-per-correct or $/accuracy claim. This corrects the ~36%
   Opus undercharge and removes the Opus/Sonnet asymmetry caused by differing
   price-card coverage.
3. **Frame the headline finding correctly and confidently:** on obvious prompts,
   Opus 4.8 adaptive thinking *correctly declines to think* even at `max` effort —
   `effort` is soft guidance, not a floor. Higher effort barely changes Opus
   tokens/cost on obvious items (low overthinking), whereas Sonnet 4.6 spends
   materially more thinking on the same obvious items (higher `overthinking_index`).
   That is a clean, defensible answer to the panel's research question — but state
   it about **billed output tokens**, not summary-count "reasoning tokens."
4. **Re-instrument before publishing a thinking curve.** True per-sample thinking
   needs `usage.output_tokens_details.thinking_tokens`, which is absent from the
   current `.eval` logs. A thinking-vs-effort curve requires a re-run with the
   provider patch in §6, or an out-of-band capture.

## 6. Concrete patches

**P1 — Cost (highest priority): price the full billed output once.**
`scripts/price_usage_with_runcost.mjs`. The carve-out depends on a per-card
reasoning rate that the Opus models-dev card lacks. `output_tokens` already
includes billed thinking and (for Anthropic & OpenAI) thinking is billed at the
output rate, so price it as a single component:

```js
// was: const outputTextTokens = Math.max(outputTokens - reasoningTokens, 0);
const outputTextTokens = outputTokens;          // already includes billed thinking
// drop the separate output_reasoning_tokens component; keep reasoningTokens in raw_usage only
```

Add a guard that warns if a nonzero reasoning quantity disappears from the priced
ledger (catches future silent card-coverage gaps). Re-run `obviousbench … rescore
--cost runcost` for the Anthropic rows after patching.

**P2 — Telemetry: read the canonical billed-thinking field.** Patch the Inspect
Anthropic provider to prefer `usage.output_tokens_details.thinking_tokens` over
re-tokenizing the summary. This needs **two** edits, because Inspect always
*streams* for thinking models (`auto_streaming` is true whenever thinking is used)
and the Anthropic SDK's streaming accumulator **drops `output_tokens_details`**
(the final `get_final_message()` / `current_message_snapshot` reports it as
`None`), even though every `message_delta` event carries it (confirmed live:
`message_delta.usage.output_tokens_details = {thinking_tokens: 23}` →
`final.usage.output_tokens_details = None`). Non-streaming `messages.create` is
unaffected and already populates the field. The two edits:
1. `_capture_compaction_from_stream` — capture `output_tokens_details` from the
   streamed `message_delta` events and restore it onto the final snapshot.
2. `model_output_from_message` — prefer the billed field, falling back to the
   summary count only when it is absent (e.g. Sonnet 3.7).
Only affects *future* runs (existing `.eval` logs never captured the field).

**P3 — Panel metadata: fix the stale display field.**
`configs/paper_v1_top_thinking_clean_20260602_panel.yaml` declares
`provider_request_settings.thinking.display: omitted`, but Inspect executes
`display: summarized`. Update to `summarized` to match the executable request (this
is what the probe's `provider_request_settings_display_mismatch` warning flags).

**P4 — Probe semantics.** `obviousbench/research/anthropic_thinking_probe.py`'s
`thinking_sparse` and `observed_reasoning_below_estimate` warnings compare
summary-length tokens against budget-token estimates (apples vs oranges). Re-baseline
against `output_tokens` or true `thinking_tokens`, and relabel the axis.

## 7. Open questions for Anthropic support (if treated as provider-side)

1. The adaptive-thinking effort table lists `max` as "Claude always thinks with no
   constraints on thinking depth," yet Opus 4.8 at `effort=max` returns a text-only
   response with `thinking_tokens=0` on trivial prompts. Is `max` a soft bias
   (consistent with the effort page's "behavioral signal, not a strict token
   budget") or is "always thinks" a guarantee? Clarify the intended floor, if any.
2. Confirm `usage.output_tokens_details.thinking_tokens` is the canonical
   billed-thinking field for Opus 4.8/4.7 and is populated under both
   `display:summarized` and `display:omitted`. **SDK bug to report:** under
   streaming, the Python SDK's accumulated final message
   (`get_final_message()` / `current_message_snapshot`) returns
   `usage.output_tokens_details = None` even though the `message_delta` events
   carry `output_tokens_details.thinking_tokens`. The accumulator should merge
   that subfield (we work around it locally in P2 edit 1).
3. Confirm that summarized thinking text is generated by a separate model and is
   **not** bounded by the original thinking length (we observe summary 63 tok vs
   billed 23 tok on Opus 4.8), i.e. summary length must never be used as a billing
   or effort proxy.
4. Confirm the runcost/price-card expectation: Anthropic bills internal thinking at
   the standard output-token rate (no separate reasoning SKU), so `output_tokens` is
   the sole authoritative basis for output-side cost.

- The 11 Anthropic Opus/Sonnet entries in the 222-panel

  **`control_style: null`** and passed bare `effort` (not `reasoning_effort`).
